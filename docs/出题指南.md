# AgentGenesis 出题完全指南

> 本文档面向**题目作者（Problem Author）**，旨在提供一份零遗漏、端到端的出题规范说明。AgentGenesis 的评测生态能否繁荣，完全取决于题目是否设计得清晰、可测、可复现。请仔细阅读每一个字段、每一种模式、每一处陷阱。

---

## 一、引言：什么是"出题"

在 AgentGenesis 中，"出题"意味着你作为第三方开发者，设计一个**可以被 LLM Agent 解决的任务**，并提供：

1. **任务描述**（Agent 要做什么）
2. **交互接口**（Agent 能调用什么工具/函数）
3. **判分逻辑**（怎样算对、怎样算错、得多少分）
4. **测试用例**（多少个 case、每个 case 的数据如何生成）
5. **运行环境**（依赖哪些 Python 包、需要多少资源）

平台会把你的代码（Judge 端）和用户提交的代码（User 端）分别放进**两个隔离的 Docker 沙箱**，通过 gRPC `SandboxBridge` 协议交换 JSON 消息，从而完成评测。

---

## 二、出题目录结构总览

一个标准题目目录如下所示：

```
problems/<problem_name>/
├── config.py                     # 题目配置（PhaseConfig 子类）
├── register.py                   # 注册/发布脚本（非必须，但强烈推荐）
├── sandbox/
│   ├── run.py                    # Judge 入口，必须存在（或配置 artifact_entry）
│   ├── user_adapter.py           # 用户侧 API 适配器
│   ├── environment.py            # 环境状态机与计分逻辑（强烈建议分离）
│   ├── generator.py              # 测试用例生成器
│   ├── data/                     # 静态数据（可选）
│   │   └── ...                   # 可被 image_data_dirs 预烘焙到镜像
│   └── visibility_manifest.json  # 私密文件清单（由 SDK 自动生成，不要手写）
└── problems/                     # 本地测试用例（可选，供本地调试）
    └── case_*.json
```

**关键约定：**
- `config.py` 中定义的类必须**继承 `PhaseConfig`**。
- `sandbox/run.py` 是 Judge 的入口点，默认路径为 `sandbox/run.py`。
- `sandbox/user_adapter.py` 必须实现 `UserAdapter` 协议（`create_user_api` 方法）。
- 所有依赖通过 `pip_dependencies` 声明，会在沙箱启动时 `uv pip install`。

## 三、PhaseConfig 字段全解

`PhaseConfig` 是 Pydantic `BaseModel` 子类，定义了一个**阶段（Phase）**的所有运行参数。每个题目至少有一个 Phase（`phase_order=1`），复杂题目可以有多个 Phase。

以下按**功能分组**，列出**每一个字段**的名称、类型、默认值、说明和平台行为。

### 3.1 基础元数据

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `phase_name` | `str` | `""` | 阶段名称，展示给用户。如 `"Basic Maze"`、`"Phase 1: 2PC"`。 |
| `phase_type` | `str` | `"agent"` | 阶段类型。目前只支持 `"agent"`（未来可能扩展为 `"code"` 等）。 |
| `phase_order` | `int` | `1` | 阶段顺序。一个 Problem 可以有多个 Phase（1, 2, 3...），按此顺序执行。 |
| `phase_level` | `str` | `"Easy"` | 难度等级。可选 `"Easy"` / `"Medium"` / `"Hard"`。仅用于展示。 |
| `description` | `str` | `""` | **阶段详细描述**。支持 Markdown。这是用户看到的"题目面"，务必详细说明接口、规则、评分、陷阱。 |
| `starter_code` | `str` | `""` | **起始代码**。用户打开题目时看到的默认 `solution.py` 骨架。通常包含函数签名和 docstring。 |

**最佳实践：**
- `description` 必须包含：1) 接口说明 2) 规则 3) 评分方式 4) 常见陷阱。
- `starter_code` 务必可直接运行（哪怕只是 `pass`），降低用户上手门槛。

### 3.2 评测器选型（决定运行模式）

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `evaluator_module` | `str` | `""` | 评测器模块路径。单 Agent 用 `"agent_genesis.dual_sandbox_evaluator"`，多 Agent 用 `"agent_genesis.isolated_evaluator"`。 |
| `evaluator_class` | `str` | `""` | 评测器类名。单 Agent 用 `"DualSandboxEvaluator"`，多 Agent 用 `"IsolatedMultiAgentEvaluator"`。 |

**两种模式的核心差异：**

| 模式 | Evaluator | 沙箱数量 | 适用场景 |
|------|-----------|----------|----------|
| **单 Agent (Pair)** | `DualSandboxEvaluator` | 1 Judge + 1 User | 大多数题目：结构化输出、迷宫、爬虫、日志分析 |
| **多 Agent (Isolated)** | `IsolatedMultiAgentEvaluator` | 1 Judge + N Agent | 多角色博弈/协作：狼人杀、微服务雪崩 |

### 3.3 测试用例与通过条件

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `num_cases` | `int` | `10` | 总测试用例数。 |
| `min_passed_cases` | `Optional[int]` | `None` | 最少通过 case 数。**`None` 表示必须全部通过**。如 `min_passed_cases=2` 表示通过 2 个即算本阶段通过。 |
| `parallel_cases` | `int` | `1` | 并发执行的 case 数。`1` 为串行，`>1` 时多个 case 同时跑（各自独立沙箱）。 |

**注意：**
- `parallel_cases` 增加时，资源消耗线性增长。确保 Worker 节点有足够容器配额。
- `min_passed_cases` 常用于"部分通过即算过"的场景（如狼人杀：3 局赢 2 局即可）。

### 3.4 资源与时间限制

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `sandbox_cpu_count` | `int` | `0` | 沙箱 CPU 核数。`0` 表示不限制。 |
| `memory_limit_mb` | `int` | `0` | 内存限制（MB）。`0` 表示不限制。超出后触发 MLE（Memory Limit Exceeded）。 |
| `sandbox_timeout` | `int` | `400` | **整个阶段的总超时**（秒）。包括所有 case + 环境初始化 + pip 安装。 |
| `case_idle_timeout` | `int` | `300` | **单个 case 的 idle 超时**（秒）。如果 Agent 在某一步超过此时间未响应，判为 TLE。 |
| `user_deps_timeout` | `int` | `120` | `uv pip install` 超时（秒）。依赖过多时需调大。 |
| `chmod_timeout` | `int` | `10` | 文件权限设置超时（秒）。一般无需修改。 |
| `run_timeout` | `int` | `180` | 单个 case 的运行超时（秒）。Judge 侧单 case 执行上限。 |
| `time_limit` | `float` | — | **题目自定义的时间限制**（秒）。由 `run.py` 自行读取并强制执行。不是平台强制字段，但几乎所有题目都会用它。 |

**时间限制的关系：**
```
sandbox_timeout  >  num_cases × max(case_idle_timeout, run_timeout, time_limit)
```
- `case_idle_timeout`：Agent 两步之间发呆的最长时间。
- `run_timeout`：Judge 端单个 case 的硬上限。
- `time_limit`：题目逻辑层面的时间预算（如"必须在 27 秒内提交"）。

**MLE 检测逻辑：**
- 内存使用 ≥ 限制值的 `98%` → 直接判 MLE。
- 内存使用 ≥ 限制值的 `85%` **且** stderr 出现 `"memoryerror"` / `"out of memory"` / `"std::bad_alloc"` → 判 MLE。

### 3.5 依赖与包管理

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `pip_dependencies` | `list[str]` | `[]` | Judge 端需要 `pip install` 的包列表。如 `["openai", "pydantic"]`。 |
| `allowed_packages` | `list[str]` | `[]` | **白名单包名**。如果非空，`pip_dependencies` 中的每个包必须经过白名单校验（不区分大小写、忽略 `-` `.` `_` 差异）。 |
| `image_data_dirs` | `list[str]` | `[]` | 需要预烘焙到 Docker 镜像的数据目录列表。如 `["data"]` 表示把 `sandbox/data/` 下的文件在镜像构建阶段 COPY 进去，避免每个 case 都重新传输。 |

**白名单校验示例：**
```python
allowed_packages = ["openai", "pydantic", "requests"]
pip_dependencies = ["openai>=1.0", "pydantic==2.5", "numpy"]
# 结果：numpy 不在白名单，注册时抛出 ValueError
```

**image_data_dirs 的作用：**
- 大数据集（如 1000 道结构化输出题、日志文件）如果通过 `write_files_chunked` 每个 case 都写一遍，会严重拖慢启动。
- 设置 `image_data_dirs=["data"]` 后，`TemplateImagePool` 会在构建镜像时把 `sandbox/data/` 的内容 COPY 到基础镜像层中，后续 case 启动直接复用该镜像，实现"冷启动加速"。
- `image_data_dirs` 的哈希是 `TemplateImagePool` 缓存键的一部分：如果数据内容变了，会自动构建新镜像。

### 3.6 Artifact 与入口点

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `artifact_url` | `str` | `""` | Artifact 的远程 URL（云端模式）。本地出题无需填写。 |
| `artifact_checksum` | `str` | `""` | Artifact 的 SHA256 校验和。 |
| `artifact_size` | `int` | `0` | Artifact 大小（字节）。 |
| `artifact_entry` | `str` | `""` | Judge 入口脚本路径。默认 `"sandbox/run.py"`。 |
| `artifact_base64` | `str` | `""` | Artifact ZIP 的 Base64 编码。由 `register.py` 中的 `build_artifact_from_dir` 生成。 |

**Artifact 是什么？**
- Artifact 是一个 ZIP 文件，包含 `sandbox/` 目录下的所有文件（以及可能的其他目录如 `wolf_agent/`）。
- 云端评测时，Worker 会下载 Artifact 并解压到 Judge 沙箱的 `/workspace/judge/` 中。
- 本地评测时，`LocalArtifactBuilder` 直接读取本地文件系统。

**默认入口点解析优先级：**
1. `config.artifact_entry`（如 `"sandbox/run.py"`）
2. 若未设置，回退到 `"sandbox/run.py"`
3. 若不存在，回退到 `"sandbox/main.py"`
4. 若仍不存在，抛错

### 3.7 用户侧入口与适配器

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `solve_attr_name` | `str` | `"solve"` | 用户 `solution.py` 中的入口函数名。如 `"solve"`、`"solve_witch"`。 |
| `adapter_preset` | `str` | `""` | Adapter 预设名。`user_adapter.py` 中的 `get_adapter(preset_name)` 会收到这个值。用于区分不同题目或不同模式。 |
| `user_bridge` | `str` | `""` | 自定义 user bridge 代码（高级）。一般留空，由评测器自动生成。 |

**单 Agent 模式：**
- `solve_attr_name="solve"`：评测器会调用 `solution.solve(env)`，其中 `env` 由 `user_adapter.py` 创建。

**多 Agent 模式（Isolated）：**
- `solve_attr_name` 可以是任意值（如 `"solve_witch"`），但真正的入口由 `solve_entry_map` 决定。
- 每个 Agent 沙箱会收到 `AGENT_ID` 环境变量，对应 `agent_ids` 中的一项。

### 3.8 私密文件与可见性

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `private_files` | `Optional[list[str]]` | `None` | **私密文件列表**。这些文件在打包 Artifact 时会被标记为 `private`，用户沙箱**无法看到**。`None` 表示不注入 visibility manifest（仅内部 Worker 可用）；`[]` 表示所有文件公开。 |

**private_files 的行为：**
- `None`：不生成 `visibility_manifest.json`。适用于内部系统题目，Contributors 无法修改 Artifact。
- `[]`（空列表）：生成 manifest，但所有文件都是公开的。用户沙箱能看到全部内容。
- `["data", "sandbox/data_pool.json"]`：这些路径对应的文件对用户不可见。

**重要：** `private_files` 只在**云端打包 Artifact** 时生效。本地调试时，用户代码和 Judge 代码都在同一台机器，无法真正隐藏文件。防作弊主要依赖云端双沙箱隔离。

### 3.9 Gateway LLM 配额

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `gateway_max_chars` | `int` | `5000000` | 该阶段所有 case 的 LLM 总字符配额（通过 Gateway 代理计费）。 |
| `gateway_max_requests` | `int` | `1000` | 该阶段所有 case 的 LLM 总请求数配额。 |
| `gateway_ttl_minutes` | `int` | `30` | Gateway Token 有效期（分钟）。 |
| `gateway_allowed_models` | `list[str]` | `[]` | 允许调用的模型白名单。空列表表示不限制。 |
| `allow_user_key` | `bool` | `False` | 是否允许用户使用自己的 LLM Key（绕过 Gateway）。`False` 表示强制走 Gateway 计费。 |

**配额计算逻辑：**
- 云端创建 Token 时，如果 `aggregate_limit=True`（默认），则实际限额 = `gateway_max_XXX × num_cases`。
- 例如 `num_cases=10, gateway_max_requests=100` → Token 允许 1000 次请求。
- `attach_llm_usage_delta` 在每个 case 结束后计算增量，更新 usage。

**Judge 端默认模型映射：**
平台根据用户选择的 Key Manufacturer，自动给 Judge 端注入默认模型：
- `openai` → `gpt-5.2`
- `deepseek` → `deepseek-chat`
- `anthropic` → `claude-4.6-sonnet`
- `qwen` → `qwen-plus`
- ...（见 `runtime/sandbox.py` 中的 `_JUDGE_MANUFACTURER_MODEL`）

### 3.10 环境变量注入

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `judge_envs` | `dict[str, str]` | `{}` | 额外注入到 Judge 沙箱的环境变量。也会注入到 User 沙箱（因为 `build_user_envs` 同样读取此字段）。 |

**自动注入的环境变量（无需手动设置）：**
- `SUBMIT_ID`, `PHASE_ID`, `USER_ID`
- `PHASE_CONFIG`：当前 PhaseConfig 的 JSON 序列化（`run.py` 通过 `os.getenv("PHASE_CONFIG")` 读取自定义字段）
- `RUNTIME_CONFIG`：RuntimeConfig 的 JSON 序列化
- `PYTHONPATH`：`/workspace/judge`（Judge）或 `/workspace/user`（User）
- `DUAL_SANDBOX_MODE`：`"judge"` 或 `"user"`
- `PYTHONUNBUFFERED`：`"1"`
- 各种 `OPENAI_API_KEY`、`DEEPSEEK_API_KEY`...（通过 Gateway Token 统一注入）

### 3.11 代码大小限制

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `max_code_size` | `int` | `1000000` | 用户代码最大大小（字节）。超过则拒绝提交。 |

## 四、ProblemConfig 字段全解

`ProblemConfig` 是题目级别的配置，一个 Problem 可以包含多个 Phase。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `title` | `str` | — | **题目名称**（必填）。长度 2-200，只能包含 ASCII 可打印字符。 |
| `overview` | `str` | `""` | 题目一句话简介。展示在题目列表页。 |
| `background` | `str` | `""` | 题目背景知识（支持 Markdown）。帮助用户理解任务领域。 |
| `level` | `str` | `"Medium"` | 题目整体难度。可选 `"Easy"` / `"Medium"` / `"Hard"`。 |
| `language` | `str` | `"en"` | 题目默认语言。 |
| `is_public` | `bool` | `False` | 是否公开题目。`False` 时仅内部可见。 |
| `data_public` | `bool` | `False` | 题目数据（测试用例）是否公开。 |
| `gitattributes` | `str` | `""` | 自定义 `.gitattributes` 内容（高级）。 |
| `phases` | `list[PhaseConfig]` | `[]` | 阶段列表。至少包含一个 Phase。 |

**slug 生成：**
- `ProblemConfig.slug` 是自动计算的：`title.lower().replace(" ", "-").replace("_", "-")`，并去除非法字符。
- Registry API 的 URL 中使用 slug，如 `/api/v1/problems/s/{slug}/revisions`。

## 五、config.py 编写规范

### 5.1 最小结构

```python
from typing import Optional
from agent_genesis import PhaseConfig

class MyProblemConfig(PhaseConfig):
    # ===== 内置字段（显式声明，防止遗漏）=====
    phase_name: str = "My Problem"
    phase_type: str = "agent"
    phase_order: int = 1
    phase_level: str = "Medium"
    num_cases: int = 10
    min_passed_cases: Optional[int] = None
    parallel_cases: int = 1
    sandbox_timeout: int = 400
    case_idle_timeout: int = 300
    pip_dependencies: list[str] = ["openai", "pydantic"]
    artifact_entry: str = "sandbox/run.py"
    solve_attr_name: str = "solve"
    adapter_preset: str = "my_problem"
    evaluator_module: str = "agent_genesis.dual_sandbox_evaluator"
    evaluator_class: str = "DualSandboxEvaluator"
    private_files: Optional[list[str]] = []

    # ===== 自定义扩展字段（供 run.py 读取）=====
    time_limit: float = 60.0
    max_attempts: int = 4
```

### 5.2 为什么要显式声明所有内置字段？

虽然 `PhaseConfig` 有默认值，但**强烈建议**在子类中显式声明所有你关心的字段。原因：
1. **自文档化**：一眼就能看出这个题目的全部配置。
2. **防遗漏**：避免继承链中某个默认值不符合预期。
3. **IDE 补全**：方便维护。

### 5.3 自定义扩展字段

任何 `PhaseConfig` 没有预定义的字段，都可以作为**自定义扩展字段**加到子类中。`run.py` 通过以下方式读取：

```python
import json, os
PHASE_CONFIG = json.loads(os.getenv("PHASE_CONFIG", "{}"))
time_limit = float(PHASE_CONFIG.get("time_limit", 60.0))
max_attempts = int(PHASE_CONFIG.get("max_attempts", 4))
```

**常见自定义字段：**
- `time_limit`：题目级时间限制
- `seed`：随机种子（`None` 表示随机）
- `max_rounds`、`max_moves`、`max_attempts`：游戏/任务上限
- `agent_ids`、`npc_agent_ids`、`solve_entry_map`：多 Agent 配置
- `num_items`、`num_questions`、`num_endpoints`：任务规模
- `tool_delay`、`info_delay`、`valid_delay`：模拟延迟

### 5.4 本地化内容变量

虽然 `ProblemConfig` 和 `PhaseConfig` 本身只有一套字段，但题目作者通常会在 `config.py` 中定义多语言内容变量，然后在 `register.py` 中选择使用：

```python
# config.py
MY_DESCRIPTION_EN = """
# Phase 1: ...
"""
MY_DESCRIPTION_ZH = """
# 第一阶段：...
"""
MY_STARTER_CODE = r'''
def solve(env):
    pass
'''
```

## 六、register.py 编写规范

### 6.1 最小结构

```python
from pathlib import Path
from agent_genesis import (
    create_problem, create_phase, init_registry,
    register_problem, sync_problem,
    build_artifact_from_dir, ClientMode, DualSandboxEvaluator,
)
from config import MyProblemConfig, MY_DESCRIPTION, MY_STARTER_CODE

def main():
    api_key = "your-api-key"
    base_url = "https://your-backend.com"
    init_registry(mode=ClientMode.USER, api_key=api_key, backend_url=base_url)

    problem_dir = Path(__file__).parent
    artifact_b64 = build_artifact_from_dir(problem_dir / "sandbox", problem_dir)

    phase = create_phase(
        DualSandboxEvaluator,
        MyProblemConfig(
            description=MY_DESCRIPTION,
            starter_code=MY_STARTER_CODE,
            artifact_base64=artifact_b64,
        ),
    )
    problem = create_problem(
        title="My Problem",
        phases=[phase],
        overview="One sentence summary.",
        is_public=True,
    )
    register_problem(problem)
    results = sync_problem(problem.title)
    print(results)

if __name__ == "__main__":
    main()
```

### 6.2 关键函数说明

#### `build_artifact_from_dir(dir_path, base_path)`
- 将 `dir_path` 目录下的所有文件打包成 ZIP，再进行 Base64 编码。
- `base_path` 决定 ZIP 中的相对路径。通常设为 `problem_dir`，这样 `sandbox/run.py` 在 ZIP 中的路径就是 `sandbox/run.py`。
- 自动处理换行符：Windows 的 `\r\n` 会被替换为 `\n`。
- 二进制文件（根据后缀判断）直接写入，文本文件统一换行符。

#### `build_artifact_from_dirs(dir_paths, base_path)`
- 多目录打包。例如狼人杀需要同时打包 `sandbox/` 和 `wolf_agent/`：
  ```python
  artifact_b64 = build_artifact_from_dirs(
      [problem_dir / "sandbox", problem_dir / "wolf_agent"],
      problem_dir,
  )
  ```

#### `create_phase(EvaluatorClass, PhaseConfig(...))`
- 将 PhaseConfig 和 Evaluator 类关联，生成一个 Phase 对象。

#### `create_problem(...)`
- 构建 ProblemConfig。

#### `register_problem(problem)`
- 将题目注册到内存中的 `ProblemRegistry`。不触发网络请求。

#### `sync_problem(title)`
- 将已注册的题目同步到云端数据库。
- **首次发布**：调用 `/api/v1/problems/register` 或 `/internal/register-problem`。
- **已存在时**：如果返回 `400` 且 body 包含 `"already published"` 或 `code == 41001`，自动回退到 **revision flow**（`/api/v1/problems/s/{slug}/revisions`）。
- **Checksum 优化**：如果 Artifact 的 SHA256 与云端已有的相同，会跳过上传，仅更新元数据。

### 6.3 发布模式

| 模式 | `ClientMode` | 端点 | 适用场景 |
|------|-------------|------|----------|
| 用户模式 | `USER` | `/api/v1/problems/register` | 普通开发者提交题目，需要审核 |
| Worker 模式 | `WORKER` | `/internal/register-problem` | 系统内部直接发布，无需审核 |

### 6.4 Revision（修订）流程

如果题目已发布，直接 `sync_problem` 会失败。此时 SDK 会自动走 **revision flow**：

1. 创建 revision（`/problems/s/{slug}/revisions`）
2. 云端返回 `auto_merged=True` → 直接生效
3. 云端返回 `auto_merged=False` → 进入待审核状态

Revision 的 payload 包含：
- `title`, `description`
- `phase_config`
- `phase_order`
- `problem_meta`（level, overview, background, is_public, data_public, gitattributes）

## 七、sandbox/ 目录详解

### 7.1 run.py —— Judge 入口与协议循环

`run.py` 是 Judge 沙箱启动后执行的第一个脚本。它不负责 gRPC 传输层（由 SDK 的 `judge_runtime.py` 提供），只负责：

1. **读取配置**（`PHASE_CONFIG` 环境变量）
2. **生成/加载测试用例**
3. **定义交互逻辑**（如何解析 Agent 的 action、如何更新环境状态）
4. **选择协议循环**（`run_turn_based_case` 或 `run_multi_agent_case`）
5. **上报结果**（`send_eval_complete`）

#### 单 Agent 模式：`run_turn_based_case`

```python
from agent_genesis.runtime.judge_runtime import JudgeRuntime, serve_judge_runtime
from agent_genesis.runtime.judge_scaffold import (
    run_case_scheduler, run_turn_based_case, send_eval_complete,
)

def run_one_case(runtime: JudgeRuntime, case_data: dict, case_index: int) -> dict:
    env = MyEnvironment(case_data)

    def _apply_action(action: dict) -> str:
        # 解析 action，更新环境，返回 observation
        return env.step(action)

    def _history_event(payload: Any) -> dict:
        # 构造 history event（会被 Router 记录，但发送给用户前会被剥离）
        return {"kind": "observation", "payload": payload, "step": env.step_count}

    def _output_data() -> dict:
        return {"success": env.success, "score": env.score}

    return run_turn_based_case(
        runtime,
        case_index=case_index,
        time_limit_seconds=TIME_LIMIT,
        get_step=lambda: env.step_count,
        apply_action=_apply_action,
        build_history_event=_history_event,
        is_done=lambda: env.done,
        is_success=lambda: env.success,
        compute_score=lambda: env.score,
        build_output_data=_output_data,
    )

def judge_main(runtime: JudgeRuntime) -> None:
    cases = generate_cases(NUM_CASES)
    results = run_case_scheduler(
        runtime,
        num_cases=len(cases),
        run_case_by_index=lambda idx: run_one_case(runtime, cases[idx], idx),
    )
    send_eval_complete(runtime, results)

def serve() -> None:
    serve_judge_runtime(judge_main)
```

**`run_turn_based_case` 参数详解：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `runtime` | `JudgeRuntime` | JudgeRuntime 实例，提供 `send_observation`、`request_action` 等方法。 |
| `case_index` | `int` | 当前 case 索引。 |
| `time_limit_seconds` | `float` | 本 case 的时间上限。 |
| `get_step` | `Callable[[], int]` | 返回当前步数（用于 history event）。 |
| `apply_action` | `Callable[[dict], str]` | 解析 Agent 发来的 action，更新环境，返回 observation 字符串。 |
| `build_history_event` | `Callable[[Any], dict]` | 构造 history event。Router 会记录这些事件用于调试，但发送给用户前会被 `sanitize_user_message` 剥离。 |
| `is_done` | `Callable[[], bool]` | 判断本 case 是否结束。 |
| `is_success` | `Callable[[], bool]` | 判断本 case 是否成功。 |
| `compute_score` | `Callable[[], int]` | 计算得分（0-100）。 |
| `build_output_data` | `Callable[[], dict]` | 构造 output_data，会随 case_result 一起返回给平台。 |

#### 多 Agent 模式：`run_multi_agent_case`

```python
from agent_genesis.runtime.multi_agent_scaffold import run_multi_agent_case

def run_one_case(runtime: JudgeRuntime, case_data: dict, case_index: int) -> dict:
    env = MyMultiAgentEnv(case_data)

    def _apply_actions(actions: dict[str, Any]) -> str:
        # actions 是 {agent_id: action_data} 的字典
        return env.apply_actions(actions)

    def _build_history_event(payload: Any) -> dict:
        return {"kind": "observation", "payload": payload}

    return run_multi_agent_case(
        runtime,
        case_index=case_index,
        agent_ids=AGENT_IDS,
        time_limit_seconds=TIME_LIMIT,
        apply_actions=_apply_actions,
        build_history_event=_build_history_event,
        is_done=lambda: env.done,
        is_success=lambda: env.success,
        compute_score=lambda: env.score,
        build_output_data=lambda: {"score": env.score},
    )
```

**多 Agent 关键差异：**
- `apply_actions` 接收的是**所有 Agent 本轮 action 的字典**，而不是单个 action。
- 每个 Agent 运行在独立沙箱中，通过 `AGENT_ID` 环境变量区分身份。
- BSP（Bulk Synchronous Parallel）协议：每轮所有 Agent 同时提交 action，Judge 统一处理后再同时下发 observation。

### 7.2 user_adapter.py —— 用户侧 API 契约

`user_adapter.py` 必须提供一个 `get_adapter(preset_name)` 工厂函数，返回 `UserAdapter` 实例。

#### 模式一：极简透传（所有交互走 Judge）

```python
import queue
from typing import Any, Optional
from agent_genesis.runtime.user_adapter import UserAdapter

class MyAdapter(UserAdapter):
    def create_user_api(self, act_queue, obs_queue):
        def _call(action_name, **kwargs):
            payload = {"action": action_name, **kwargs}
            act_queue.put(payload)
            obs = obs_queue.get()
            if obs is None:
                raise StopIteration("environment closed")
            return obs

        env = type("Env", (), {})()
        env.get_problem = lambda: _call("get_problem")
        env.submit = lambda answer: _call("submit", answer=answer)
        return env

def get_adapter(preset_name: str = "default") -> UserAdapter:
    return MyAdapter()
```

#### 模式二：混合 Adapter（本地模拟 + Judge 交互）

这是高级且常用的模式。某些操作（如耗时 IO）在 User 沙箱本地执行，避免阻塞 Judge 队列：

```python
class HybridAdapter(UserAdapter):
    def create_user_api(self, act_queue, obs_queue):
        # 本地状态（每个 User 沙箱独立）
        cached_catalog = {}

        def _call_judge(action_name, **kwargs):
            act_queue.put({"action": action_name, **kwargs})
            return obs_queue.get()

        def get_problem():
            raw = _call_judge("get_problem")
            data = json.loads(raw)
            cached_catalog.update(data["catalog"])
            return data["user_message"]

        def get_info(item_key):
            # 本地执行，不经过 Judge！
            time.sleep(20)  # 模拟延迟
            return cached_catalog.get(item_key)

        def submit_answer(price, brand):
            return _call_judge("submit_answer", price=price, brand=brand)

        env = type("Env", (), {})()
        env.get_problem = get_problem
        env.get_info = get_info
        env.submit_answer = submit_answer
        return env
```

**为什么需要混合模式？**
- `run_turn_based_case` 是**串行**的：Agent 发一个 action，Judge 处理，返回 observation，Agent 才能发下一个。
- 如果 Agent 需要并发调用多个慢工具（如 `get_weather` 和 `get_humidity`，各 10 秒），串行需要 20 秒，而本地并行只需 10 秒。
- 通过在 `user_adapter.py` 中本地模拟这些工具，Agent 可以用 `asyncio.gather` 或 `threading` 并发执行。

**注意：** 本地执行的工具**不能依赖 Judge 端的状态**，只能依赖从 Judge 一次性获取的静态数据（如预生成的天气文本）。

#### 模式三：多 Agent Isolated Adapter

多 Agent 题目中，每个 Agent 的 `user_adapter.py` 必须继承 `IsolatedAgentAdapter`（或手动读取 `AGENT_ID`）：

```python
from agent_genesis.runtime.isolated_adapter import IsolatedAgentAdapter

class WerewolfAdapter(IsolatedAgentAdapter):
    def _build_env(self, agent_id, call_tool):
        env = type("Env", (), {})()
        env.player_id = agent_id
        env.connection = lambda: call_tool("connection")

        if agent_id in ["wolf_1", "wolf_2"]:
            env.kill = lambda target: call_tool("kill", target=target)
        elif agent_id == "witch":
            env.save = lambda: call_tool("save")
            env.poison = lambda target: call_tool("poison", target=target)
        # ... 其他角色

        return env

def get_adapter(preset_name: str = "isolated_werewolf") -> UserAdapter:
    return WerewolfAdapter()
```

**关键点：**
- `call_tool(action_name, **kwargs)` 会发送 action 并阻塞等待 observation。
- 不同 `agent_id` 看到的 `env` 接口不同（角色差异化 API）。
- `AGENT_ID` 环境变量由 `IsolatedMultiAgentEvaluator` 在创建沙箱时注入。

### 7.3 environment.py —— 环境状态与计分

**强烈建议**将游戏/任务逻辑从 `run.py` 中分离到 `environment.py` 中。这样：
1. `run.py` 只负责协议循环（关注点分离）。
2. `environment.py` 可以独立单元测试。
3. 逻辑更清晰，不易出错。

#### 环境类设计模式

```python
class MyEnvironment:
    def __init__(self, case_data: dict):
        self.state = case_data["initial_state"]
        self.step_count = 0
        self.done = False
        self.success = False
        self.score = 0

    def step(self, action: dict) -> str:
        """执行一步，返回 observation 字符串。"""
        self.step_count += 1
        # ... 更新状态 ...
        if self._check_success():
            self.done = True
            self.success = True
            self.score = 100
        elif self._check_failure():
            self.done = True
            self.success = False
        return "observation text"

    def compute_score(self, optimal_value=None) -> int:
        """计算最终得分。"""
        if not self.success:
            return 0
        # 可根据效率、步数等加权
        return max(0, 100 - self.step_count)
```

**计分约定：**
- 得分范围：`0 - 100`。
- `100`：完美通过。
- `0`：失败或未通过。
- 中间分：部分正确（如狼人杀中好人赢 100、狼人赢 50）。

### 7.4 generator.py —— 测试用例生成

```python
import random

def generate_cases(num_cases: int, seed: Optional[int] = None) -> list[dict]:
    rng = random.Random(seed)
    cases = []
    for i in range(num_cases):
        case = {
            "case_index": i,
            "initial_state": rng.randint(0, 100),
            "target": rng.choice(["A", "B", "C"]),
        }
        cases.append(case)
    return cases
```

**防作弊设计：**
- **随机种子**：`seed=None` 时每次运行都生成不同的 case。云端 `run.py` 通常不传 seed。
- **语义随机化**：环境返回的文本使用多套模板随机选择（如迷宫的 "撞墙了" / "鼻子都撞酸了"），防止用户写死正则匹配。
- **私有数据池**：把题库/数据集放在 `private_files` 中，用户无法直接读取。

### 7.5 data/ 与 image_data_dirs

如果题目需要大量静态数据（如图片、日志、题库）：

1. 把数据放在 `sandbox/data/` 中。
2. 在 `config.py` 中设置 `image_data_dirs=["data"]`。
3. 如果需要隐藏，设置 `private_files=["data"]`。

**打包行为：**
- `build_artifact_from_dir` 会递归包含 `sandbox/data/` 中的所有文件。
- 云端 `extract_image_data_files` 会从 Artifact 中提取 `image_data_dirs` 对应的文件。
- `compute_data_content_hash` 计算数据内容的 SHA256，作为 `TemplateImagePool` 的缓存键。

### 7.6 visibility_manifest.json 与 private_files

`private_files` 在 `ProblemRegistry._prepare_phase_data` 中的处理逻辑：

```python
private_files = phase_data.pop("private_files", None)
if artifact_b64 and private_files is not None:
    phase_data["artifact_base64"] = inject_visibility_manifest(artifact_b64, private_files)
```

`inject_visibility_manifest` 会在 ZIP 中插入一个 `visibility_manifest.json`：
```json
{
  "private": ["data", "sandbox/secret.txt"]
}
```

云端 Worker 在解压 Artifact 到 User 沙箱时，会根据此 manifest **跳过 private 文件**。

**注意：**
- `private_files=None`：不注入 manifest。这意味着**无法通过 Contributor 流程修改 Artifact**，仅限内部系统使用。
- `private_files=[]`：注入空 manifest，所有文件对用户可见。
- `private_files=["data"]`：用户看不到 `data/` 目录。

## 八、Evaluator 选型详解

### 8.1 DualSandboxEvaluator（单 Agent / Pair 模式）

```
Worker
  ├─ create Judge Sandbox (judge_envs)
  ├─ create User Sandbox (user_envs)
  ├─ write artifact files → /workspace/judge/
  ├─ write user files → /workspace/user/
  ├─ write bridge support (eval_runtime/) → both
  ├─ uv pip install requirements.txt
  ├─ start Judge process (python -u sandbox/run.py)
  ├─ start User process (python -u solution.py bridge)
  ├─ run_pair_protocol_router
  │    ├─ case_start → restart_user_runtime
  │    ├─ observation → send_to_user
  │    ├─ action_request → request_user_action → send_to_judge
  │    ├─ case_end → parse result
  │    └─ eval_complete → finish
  └─ cleanup (destroy sandboxes)
```

**适用场景：**
- 结构化输出、迷宫、天气查询、爬虫、日志分析、工具创建等绝大多数题目。

### 8.2 IsolatedMultiAgentEvaluator（多 Agent 模式）

```
Worker
  ├─ create Judge Sandbox
  ├─ for each agent_id in config.agent_ids:
  │     ├─ determine code source:
  │     │     NPC (in npc_agent_ids) → from artifact (npc_code_prefix)
  │     │     User → from submission (user_files_bytes)
  │     ├─ create Agent Sandbox
  │     └─ inject AGENT_ID=agent_id
  ├─ run_multi_agent_case
  │    ├─ case_start{agent_ids}
  │    ├─ per round:
  │    │     action_request → fan-out to all agents
  │    │     collect actions (parallel)
  │    │     apply_actions{agent_id: action}
  │    │     build observation + history events
  │    ├─ case_end{status, score}
  │    └─ eval_complete
  └─ cleanup
```

**配置要求（额外字段）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `agent_ids` | `list[str]` | 所有 Agent 的身份标识列表。 |
| `npc_agent_ids` | `list[str]` | 由系统/NPC 控制的 Agent 子集。 |
| `npc_code_prefix` | `str` | NPC 代码在 Artifact 中的路径前缀。如 `"wolf_agent/"`。 |
| `solve_entry_map` | `dict[str, str]` | Agent ID → 入口函数名的映射。如 `{"witch": "solve_witch"}`。 |

**NPC 代码打包：**
- 狼人杀的 `wolf_agent/solution.py` 需要被打包进 Artifact。
- 使用 `build_artifact_from_dirs([sandbox_dir, wolf_agent_dir], problem_dir)` 同时打包多个目录。

## 九、高级模式与技巧

### 9.1 混合 Adapter（本地模拟 + Judge 交互）

**典型题目：** `parallel_weather`、`sports_shopping`、`short_circuit_scraper`

**核心设计：**
1. Judge 在 `case_start` 或 `get_problem` 时，一次性把**全部静态数据**传给 User（如所有城市的天气文本、商品目录）。
2. User 端的 `user_adapter.py` 把数据缓存到本地变量中。
3. 后续查询（`get_weather`、`get_info`）直接从本地缓存读取，并通过 `time.sleep` / `asyncio.sleep` 模拟延迟。
4. 提交答案时（`submit_answer`）再与 Judge 交互。

**好处：**
- Agent 可以并发调用工具（`asyncio.gather`），不受 Judge 串行协议限制。
- 减少 Judge-User 之间的 RPC 往返次数。

**注意：**
- 本地模拟的工具**必须是无状态且幂等的**，因为 Agent 可能重试、乱序调用。
- 不要在本地模拟中泄露正确答案。

### 9.2 时间控制与超时

**多层时间防线：**

1. **平台层**：`case_idle_timeout`（Agent 发呆时间）、`run_timeout`（Judge 单 case 上限）、`sandbox_timeout`（整体上限）。
2. **Judge 逻辑层**：`run_turn_based_case` 的 `time_limit_seconds` 参数，内部用 `time.time()` 检查。
3. **环境层**：`environment.py` 自己维护计时器，在 `submit_answer` 时检查是否超时。

**示例（ sports_shopping 的严格时间窗）：**
```python
# environment.py
self.problem_start_time = time.time()

def submit_answer(self, price, brand):
    elapsed = time.time() - self.problem_start_time
    if elapsed > SUBMIT_TIME_LIMIT:
        return {"error": "timeout", "score": 0}
    # ... 校验答案 ...
```

### 9.3 私有数据与防作弊

**策略矩阵：**

| 策略 | 实现方式 | 代表题目 |
|------|----------|----------|
| 私有文件 | `private_files=["data"]` | structured_output、resilient_scraper |
| 随机生成 | `seed=None`，每次不同 | maze、parallel_weather |
| 语义随机化 | 多套模板随机选择反馈文本 | maze、sports_shopping |
| 在线验证 | 答案只有 Judge 知道 | 所有题目 |
| 延迟提交 | 必须调用 `submit` 才算，不能直接从返回推断 | tool_creator_challenge |

### 9.4 TemplateImagePool 与数据预烘焙

**image_data_dirs 的使用步骤：**

1. `config.py` 中声明：
   ```python
   image_data_dirs: list[str] = ["data"]
   ```
2. 把大数据集放在 `sandbox/data/` 中。
3. `build_artifact_from_dir` 会自然包含这些文件。
4. 云端 `DualSandboxEvaluator._run_single_case`：
   - 调用 `extract_image_data_files(artifact_files, ["data"])` 提取数据文件。
   - 计算 `compute_data_content_hash(data_files)`。
   - 调用 `get_or_create_template(..., data_files=data_files, data_content_hash=data_hash)`。
5. `TemplateImagePool`：
   - 如果已有镜像的 key 匹配（pip + data + base_image），直接复用。
   - 否则构建新镜像：在基础镜像上 COPY 数据文件，再 `uv pip install`。
6. 后续 case 启动时，直接使用该镜像，无需重新写数据文件。

### 9.5 Gateway 配额配置

**为题目选择合适的配额：**

| 题目类型 | 推荐 `gateway_max_requests` | 推荐 `gateway_max_chars` | 说明 |
|----------|---------------------------|------------------------|------|
| 简单结构化输出 | 50-100 | 500K-1M | Agent 调用次数少 |
| 交互式迷宫 | 200-500 | 2M-5M | 多轮对话，步数多 |
| 狼人杀 | 1000-2000 | 50M+ | 6 个 Agent 多轮发言 |
| 日志分析 | 50-100 | 10M+ | 单次 prompt 可能很大 |
| 微服务雪崩 | 200-500 | 2M-5M | 3 个 Agent 协作 |

**allow_user_key：**
- `False`（默认）：强制所有 LLM 调用走 Gateway，平台可以精确计费、限流、审计。
- `True`：允许用户把自己的 API Key 注入环境变量，绕过 Gateway。适用于需要特殊模型或高速调用的场景，但平台无法监控用量。

以下是 `problems/` 目录下所有题目的独特设计亮点，供出题时参考。

### 1. werewolf（狼人杀）—— 多 Agent 社交推理
- **模式**：`IsolatedMultiAgentEvaluator`，6 个 Agent（2 NPC 狼 + 4 用户角色）。
- **亮点**：
  - 使用 `npc_code_prefix` + `wolf_agent/` 打包 NPC 代码。
  - 每个角色 API 不同（`kill` / `save` / `poison` / `check` / `speak` / `vote`）。
  - 严格阶段控制：非当前角色必须 `connection()`，否则判异常。
  - `min_passed_cases=2`：3 局赢 2 局即可通过。
  - 投票规则复杂（第一天可弃票，第二天强制投票）。

### 2. structured_output —— 结构化输出校验
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - `image_data_dirs=["data"]` + `private_files=["data"]`：1000 道题库预烘焙且对用户隐藏。
  - Judge 解析双方 JSON 后做**语义等价比较**（不看格式化差异）。
  - 每 case 只抽取一题，`min_passed_cases=None`（全对才过）。

### 3. parallel_weather —— 并发工具调用
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - **混合 Adapter 典范**：`get_weather`（同步本地 sleep）+ `get_humidity`（异步本地 sleep）。
  - Judge 一次性下发全部 200 个城市的预生成文本，Agent 本地并发查询。
  - `tool_delay=10.0`，`max_allowed_time=27.0`：串行必超时，必须并发。
  - 自然语言返回值中包含噪声数字，防止硬编码正则。

### 4. sports_shopping —— 守卫与延迟决策
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - 混合 Adapter：`get_info` 本地执行（20 秒延迟），`guardrail` / `submit_answer` 走 Judge。
  - **时间陷阱**：Guardrail 类 case 必须在 7 秒内响应，调用 `get_info` 会浪费 20 秒直接超时。
  - 商品描述是自然语言，Agent 需要先判断是否需要查询。

### 5. maze —— 交互式探索与语义随机化
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - 极简 API：只暴露 `move(direction)` 一个函数（而非 namespace）。
  - 反馈文本完全语义随机化（"鼻子撞酸了" / "乌鸦叫了一声"），杜绝正则作弊。
  - `generator.py` 用 Prim 算法生成迷宫 + BFS 计算最优路径。
  - 得分与步数挂钩：`optimal_moves` 作为基准计算效率分。

### 6. short_circuit_scraper —— 并行短路+取消
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - 本地多线程模拟 10 个并行 scraper，使用 `threading.Event` 实现 `cancel()` 级联终止。
  - **资源泄漏检测**：`submit` 后会等待 25 秒，统计有多少线程在提交后才完成，泄露则失败。
  - 两阶段 done 协议：`submit` 不直接结束，`cascade_check` 才最终确认。

### 7. resilient_scraper —— 指数退避重试
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - `private_files=["sandbox/data_pool.json"]`：事实库对用户隐藏。
  - 成功概率逐次提升：30% → 50% → 80% → 95%。
  - Judge 端严格校验**指数退避时间间隔**（误差容忍 1 秒）。
  - 全部失败时提交空字符串也算正确（优雅降级）。

### 8. tool_creator_challenge —— 动态工具创建
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - 答案是大整数（数百位），LLM 无法直接猜出，必须写 Python 代码计算。
  - `private_files` 包含 generator/environment/run.py，防止用户提前看到题库和判分逻辑。
  - 100 道计算题（10 种类型 × 10 道），覆盖 fibonacci、阶乘、Catalan 数等。

### 9. log_hunter —— 大规模语义理解
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - 约 800K tokens（3.2M 字符）的日志，远超普通上下文。
  - 黑客日志由 LLM 动态生成（荒诞理由），与正常日志混合。
  - `shared_user_pool`：黑客和正常用户共享用户名池，防止通过用户名模式识别。
  - 时间与准确率双重评分：全对基础分 100，耗时越短分越高。

### 10. interrupt_judge —— 批量分类准确率
- **模式**：`DualSandboxEvaluator`，单 Agent。
- **亮点**：
  - 极简配置：`pip_dependencies=[]`（无需 LLM，纯规则/统计即可）。
  - 每 case 500 条中文语料，准确率要求 **≥ 98.5%**。
  - 本地预标注数据 `data/data.jsonl`，判分完全离线。
  - `parallel_cases=5`：5 个 case 并发跑。

### 11. microservice_avalanche —— 分布式事务与混沌工程
- **模式**：`IsolatedMultiAgentEvaluator`，3 个 Agent（Order/Inventory/Payment）。
- **亮点**：
  - 完整的 2PC 状态机（INIT → PREPARED → COMMITTED / ROLLED_BACK）。
  - 混沌注入：20% 资源故障 + 10% 网络延迟/丢包。
  - **部分提交 = 0 分**：ACID 一致性是硬性要求。
  - `send_rpc` 模拟跨服务通信，消息可能延迟 1-3 轮或丢失。
  - 严格状态校验：未 PREPARE 就 COMMIT 会报错。

## 十一、常见问题与排错

### Q1: 注册时提示 pip_dependencies 不在 allowed_packages 中
**原因**：`allowed_packages` 非空，但 `pip_dependencies` 中有包不在白名单。
**解决**：要么把包加入 `allowed_packages`，要么把 `allowed_packages` 设为空列表（关闭校验）。

### Q2: 用户沙箱看不到 data/ 目录
**原因**：`private_files` 包含了 `"data"`，但 `build_artifact_from_dir` 的 `base_path` 设置不对，导致 ZIP 中路径不匹配。
**解决**：确保 `base_path` 是 `problem_dir`，这样 `sandbox/data/` 在 ZIP 中的路径就是 `sandbox/data/`。`private_files` 中写 `"data"` 时，实际匹配的是 ZIP 根目录下的 `data/`；如果 ZIP 结构是 `sandbox/data/`，则 `private_files` 应写 `["sandbox/data"]`。

### Q3: 多 Agent 题目中某个 Agent 没收到 observation
**原因**：Agent 沙箱启动失败，或 `AGENT_ID` 环境变量未正确注入。
**排查**：
- 检查 `agent_ids` 是否包含该 Agent。
- 检查 `npc_code_prefix` 对应的文件是否确实在 Artifact 中。
- 检查 `solve_entry_map` 是否映射了正确的入口函数。

### Q4: 本地能跑，云端报错 "module not found"
**原因**：`pip_dependencies` 漏写了依赖，或云端 `TemplateImagePool` 缓存了旧镜像。
**解决**：
- 确认 `pip_dependencies` 完整。
- 修改 `image_data_dirs` 或 `pip_dependencies` 中任意一项，使缓存键变化，强制重建镜像。

### Q5: Gateway Token 限额不足
**原因**：`gateway_max_requests` 或 `gateway_max_chars` 设置太小。
**解决**：根据题目实际 LLM 调用量调大配额。注意 aggregate_limit 会自动乘以 `num_cases`。

### Q6: 如何调试 Judge 逻辑？
**方法**：
1. 使用 `LocalEvaluator` 本地运行：
   ```python
   from agent_genesis.local import LocalEvaluator
   ev = LocalEvaluator("problems/maze", "path/to/solution.py", llm_config)
   result = ev.evaluate()
   ```
2. 在 `environment.py` 和 `run.py` 中加 `print(..., flush=True)`，输出会进入 stderr，可通过 `on_user_log` / `on_judge_log` 事件查看。
3. 使用 `evaluate_stream()` + `TerminalVisualizer` 实时观察交互过程。

from agent_genesis import PhaseConfig

### register.py

from pathlib import Path
from agent_genesis import (
    create_problem, create_phase, init_registry,
    register_problem, sync_problem,
    build_artifact_from_dir, ClientMode, DualSandboxEvaluator,

from config import HelloWorldConfig, DESCRIPTION, STARTER_CODE

        mode=ClientMode.USER,
        api_key="your-api-key",
        backend_url="https://your-backend.com",

### sandbox/run.py
```python
import json, os
from agent_genesis.runtime.judge_runtime import JudgeRuntime, serve_judge_runtime
from agent_genesis.runtime.judge_scaffold import (
    run_case_scheduler, run_turn_based_case, send_eval_complete,
)
from environment import HelloWorldEnvironment
from generator import generate_cases

PHASE_CONFIG = json.loads(os.getenv("PHASE_CONFIG", "{}"))
TIME_LIMIT = float(PHASE_CONFIG.get("time_limit", 10.0))
NUM_CASES = int(PHASE_CONFIG.get("num_cases", 3))
MAX_NUMBER = int(PHASE_CONFIG.get("max_number", 100))


def parse_guess(action: dict):
    data = action.get("data", {})
    if isinstance(data, dict):
        return int(data.get("number", 0))
    return int(data)


def run_one_case(runtime: JudgeRuntime, case_data: dict, case_index: int):
    env = HelloWorldEnvironment(case_data)

    def _apply_action(action: dict) -> str:
        n = parse_guess(action)
        return env.guess(n)

    def _history_event(payload):
        return {"kind": "observation", "payload": payload, "step": env.attempts}

    return run_turn_based_case(
        runtime,
        case_index=case_index,
        time_limit_seconds=TIME_LIMIT,
        get_step=lambda: env.attempts,
        apply_action=_apply_action,
        build_history_event=_history_event,
        is_done=lambda: env.done,
        is_success=lambda: env.success,
        compute_score=lambda: 100 if env.success else 0,
        build_output_data=lambda: {"target": env.target, "attempts": env.attempts},
    )


def judge_main(runtime: JudgeRuntime):
    cases = generate_cases(NUM_CASES, max_number=MAX_NUMBER)
    results = run_case_scheduler(
        runtime,
        num_cases=len(cases),
        run_case_by_index=lambda idx: run_one_case(runtime, cases[idx], idx),
    )
    send_eval_complete(runtime, results)


def serve():
    serve_judge_runtime(judge_main)


if __name__ == "__main__":
    serve()
```

### sandbox/environment.py
```python
import random

class HelloWorldEnvironment:
    def __init__(self, case_data: dict):
        self.target: int = case_data["target"]
        self.attempts: int = 0
        self.max_attempts: int = 10
        self.done: bool = False
        self.success: bool = False

    def guess(self, n: int) -> str:
        self.attempts += 1
        if self.attempts >= self.max_attempts and n != self.target:
            self.done = True
            return f"game over. the answer was {self.target}"
        if n > self.target:
            return "too high"
        if n < self.target:
            return "too low"
        self.done = True
        self.success = True
        return "correct"
```

### sandbox/generator.py

import random

def generate_cases(num_cases: int, max_number: int = 100) -> list[dict]:
    return [

        for i in range(num_cases)
    ]

### sandbox/user_adapter.py
```python
import queue
from typing import Any, Optional
from agent_genesis.runtime.user_adapter import UserAdapter

class HelloWorldAdapter(UserAdapter):
    def create_user_api(self, act_queue, obs_queue):
        def _call(action_name, **kwargs):
            act_queue.put({"action": action_name, **kwargs})
            obs = obs_queue.get()
            if obs is None:
                raise StopIteration("environment closed")
            return obs

        env = type("Env", (), {})()
        env.get_range = lambda: _call("get_range")
        env.guess = lambda n: _call("guess", number=int(n))
        return env

def get_adapter(preset_name: str = "hello_world") -> UserAdapter:
    return HelloWorldAdapter()
```

## 结语

出题是一门**工程艺术**。一个好的 AgentGenesis 题目应该：

1. **接口极简**：Agent 只需要知道尽可能少的函数。
2. **规则明确**：description 中把边界情况、常见陷阱写清楚。
3. **评分公平**：success / score / done 逻辑无歧义。
4. **防作弊到位**：随机生成、语义随机化、私有数据三层防护。
5. **可本地调试**：`environment.py` 与 `run.py` 分离，`LocalEvaluator` 一键运行。
6. **资源合理**：时间、内存、Gateway 配额既够用又不浪费。
