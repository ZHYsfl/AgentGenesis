# AgentGenesis 项目全面总结

## 一、项目定位

AgentGenesis 是一个**面向 Agent 评测与训练的 Python SDK**,核心使命是让"出题人"和"答题人(Agent)"可以解耦协作:

- **出题人**通过 `config.py` + `sandbox/` 描述题目环境与判分逻辑,通过 `register.py` 把题目发布到云端 Registry
- **答题人**只需写一份 `solution.py`,在 `LocalEvaluator` 里本地跑通后,提交到云端 Worker 集群异步评测
- **平台方**通过统一的 gRPC `SandboxBridge` 协议、Docker 双沙箱、Gateway 计费令牌,把题目作者代码、用户代码与 LLM 计费完全隔离

它同时支持 **single-agent (PairSession)****multi-agent (IsolatedSession)** 两种范式,并提供完整的本地复现链路。

## 二、顶层目录结构

```
agent_genesis/
├── runtime/         # 评测核心:gRPC 协议、Pair/Isolated 会话、Router 状态机、Docker 沙箱与模板池
│   ├── protocol.py / router.py           # 消息枚举 + 中央状态机
│   ├── pair_session.py / isolated_session.py  # 双沙箱 / 多 Agent 会话编排
│   ├── multi_agent_scaffold.py           # BSP 屏障调度
│   ├── sandbox.py / process.py / artifact.py  # 容器、进程、构件下载
│   ├── gateway.py / history.py / results.py   # 计费、历史、结果归一化
│   ├── user_runtime.py / user_adapter.py / isolated_adapter.py  # 用户侧 gRPC bridge + 适配器协议
│   └── proto/eval_bridge.proto           # SandboxBridge 三方法
├── tool_calling/    # 异步 LLM Agent 工具调用框架(自带错误自纠)
├── local/           # 本地评测器(LocalEvaluator)与终端可视化
├── proto/           # gRPC 自动生成的 stub
└── runtime/__init__.py  # 模块对外 lazy export 入口
```

## 三、评测生命周期(全链路)

### 3.1 Worker 端(云上)

```
EvaluationService._tick
  → 轮询 pending_submission(claim 锁)
  → 收到 SIGTERM 时 unclaim 让其它 worker 接管
  → download_artifact (SHA256 + 指数退避)
  → extract_artifact (跳过 .. 路径)
  → 判定题目类型(SingleAgent / MultiAgent / Isolated)
  → 选择 DualSandboxEvaluator 或 IsolatedMultiAgentEvaluator
```

### 3.2 评测器层

```
DualSandboxEvaluator.evaluate
  ├─ resolve_sandbox_resources  (cpu/memory)
  ├─ TemplateImagePool.get_or_build
  │     key = sha256(pip_lock + image_data_dirs + base_image)
  │     LRU GC,保留 N 个缓存镜像
  ├─ create_gateway_token_for_user
  │     aggregate_limit 时把 max_chars/max_requests × num_cases
  ├─ 每个 case 串行或并发:
  │     _SandboxPairSession.run(deps: PairSessionDeps)
  ├─ revoke_gateway_token
  └─ shutdown_all_sandboxes (兜底清理)
```

### 3.3 PairSession.run

```
setup:
  judge_sb = SandboxManager.create(template, judge_envs)
  user_sb  = SandboxManager.create(template, user_envs)
  写 artifact + bridge_support + user_files (write_files_chunked, 200/批)
  uv pip install -r requirements.txt (filter_requirements 白名单)
  start_background_python (judge → user, 都重定向 stdout→stderr)
  wait_for_ready (gRPC CheckReady)

run:
  run_pair_protocol_router(deps)  ← 中央状态机

cleanup:
  失败时 tail 4096 字节 stderr
  destroy 两个容器
```

### 3.4 Router 状态机(`runtime/router.py`)

轮询 judge 端的 RecvMessage,按 `type` 分发:

| MessageType      | 处理                                                                    |
| ---------------- | ----------------------------------------------------------------------- |
| `case_start`     | restart_user_runtime,初始化 history,触发 `on_case_start`               |
| `case_request`   | 回 `case_assign{case_index}` 或 `case_stop`                            |
| `observation`    | 写历史 → sanitize(剥 `history_events`) → send_to_user                  |
| `action_request` | 向 user 拉 action,经 `_coerce_action_message` 校验,转发回 judge        |
| `case_end`       | parse_case_result → attach_llm_usage_delta → attach_case_history → on_case_end |
| `eval_complete` / `error` | break 返回 `ProtocolRunState`                                  |

`_coerce_action_message` 把用户回包统一成 `{type:"action", data|error}`,容忍超时 / OOM / 异常退出(`is_likely_mle_exit` 内存阈值识别)。

## 四、沙箱与模板池

- **DockerSandbox** 提供 ExecHandle,统一 `cd workdir && source /workspace/.venv/bin/activate && exec python -u script 2> stderr_file`
- **SandboxManager** 用信号量做整机并发上限,提供 `shutdown_all` 紧急清理
- **TemplateImagePool** 三层哈希键:
  - `pip_lock` 内容
  - `image_data_dirs`(预烘焙静态数据集,如题目大文件)
  - `base_image_identity`

  LRU 淘汰,命中即跳过 `pip install`,大幅缩短冷启动
- **资源**:`PhaseConfig.resources` → `cpu_count` / `memory_mb` 注入容器
- **MLE 探测**:容器内 ≥98% 内存 或 (≥85% + "memoryerror"/"out of memory"/"std::bad_alloc") 视作内存超限

## 五、gRPC SandboxBridge 协议

```proto
service SandboxBridge {
  rpc CheckReady(Empty) returns (ReadyStatus);
  rpc SendMessage(SendRequest)  returns (SendAck);     // worker → 容器
  rpc RecvMessage(RecvRequest)  returns (RecvResponse);// 容器 → worker
}
```

- **传输层**:50MB max msg,8 worker 线程,默认端口 50052(`SANDBOX_GRPC_PORT` 可覆盖)
- **应用层**:JSON 信封 `{type, data?, error?, status?, traceback?, ...}`
- **MessageType** 全集:`case_start / observation / action_request / action / case_end / eval_complete / error / case_request / case_assign / case_stop`,通过 `protocol._registry` 可扩展自定义类型
- **history_events**:judge 投放在 observation 上的旁路日志,送给 user 前由 `sanitize_user_message` 剥离

## 六、用户侧 Bridge 与适配器

### 6.1 `user_runtime.bridge_main`

1. 启动 `_user_thread`,内部 `with redirect_stdout(sys.stderr): entry(user_api)`(防止 print 污染协议)
2. `entry` 即 `solution.solve`,通过 `user_api`(由 adapter 提供)读 obs / 发 action
3. 异常退出时把 traceback 写入 outbound action 包

### 6.2 `UserAdapter` 双契约

- `UserAdapter` ABC:供出题人继承,IDE 自动补全
- `UserAdapterProtocol`(@runtime_checkable):用 `isinstance` 鸭子检查,**对 `eval_runtime.*` vs `agent_genesis.*` 模块路径不敏感**,解决 sandbox 内 import 路径漂移问题

### 6.3 `IsolatedAgentAdapter`(多 Agent)

- 从 `os.environ["AGENT_ID"]` 读取自身身份
- `_call_tool(action_name, **kwargs)` 把 action 入 `act_queue`,阻塞 `obs_queue.get()` 取观察(`None` → `StopIteration` 优雅终止)
- 子类实现 `_build_env(agent_id, call_tool) → SimpleNamespace`,注入题目级 API

## 七、多 Agent 调度(`multi_agent_scaffold.py`)

BSP(Bulk Synchronous Parallel)屏障:

```
case_start{agent_ids}
loop:
  judge → action_request
  worker fan-out:每个 agent 发送其 obs.data[agent_id]
  并行 fan-in:ThreadPoolExecutor.as_completed 收集每个 agent 的 action
  整合为 {type:"action", data:{agent_id: result}}
  → judge.apply_actions(actions)
  → judge 生成下一轮 obs(含 history_events)
case_end{status, score, time_used, output_data, error}
```

## 八、出题人约定(Problem Authoring)

```
problem_xxx/
├── config.py          # 继承 PhaseConfig(timeout/memory/visibility/llm_quota/...)
├── register.py        # 调 PublishClient.publish 或 publish_revision
├── sandbox/
│   ├── run.py             # judge 入口(可省,默认 sandbox/run.py)
│   ├── user_adapter.py    # 实现 create_user_api,定义题目侧 API
│   ├── data/...           # 静态资料(可声明为 image_data_dirs 烘焙到模板镜像)
│   └── visibility_manifest.json  # private_files 白名单(打 zip 时剥离)
└── problems/
    └── case_*.json    # 测试用例
```

- **Registry 发布**:先尝试 `publish`,若已存在(HTTP 41001)→ 退化为 `publish_revision`,保持版本递增
- **Artifact 打包**:把 sandbox/ 树打 zip,按 `visibility_manifest.json` 剥离 private_files(用户容器拿不到)
- **image_data_dirs**:在题目大数据场景下声明后,`TemplateImagePool` 把数据集 COPY 进基础镜像层,避免每个 case 都重新写

## 九、Gateway 计费令牌

`runtime/gateway.py` 提供 LLM 令牌中转:

- `create_gateway_token_for_user(num_cases, aggregate_limit=True)`:per-user token,limit 整体放大
- 容器环境注入统一 base_url(`SANDBOX_GATEWAY_URL/gateway/v1/keys/{key_name}`)
- `build_llm_env_vars` 把同一 token 同步到 `OPENAI_API_KEY / DEEPSEEK_API_KEY / ANTHROPIC_API_KEY / ...` 等所有主流厂商环境变量
- `_JUDGE_MANUFACTURER_MODEL` 提供 judge 端默认模型回退表(openai→gpt-5.2,deepseek→deepseek-chat,...)
- `attach_llm_usage_delta` 在每个 case 结束时计算与上一 case 的 char/request 增量,挂到 case_result.usage

## 十、tool_calling 异步 Agent 框架

### 10.1 核心模型

```python
LLMConfig(api_key, model, base_url, extra_body=None)  # model_dump 自动剔 None extra_body
Tool(name, description, function, parameters)
Agent(config, max_tool_retries=3, debug=False)        # AsyncOpenAI 客户端
```

### 10.2 `Agent._execute_tool_call` 结构化错误

四类错误带 `_tool_status`/`_error_type` 标记,而非靠字符串匹配:

- `parse_error`:JSON 参数解析失败
- `not_found`:函数名未注册
- `arg_error`:参数签名不匹配(给出 expected vs actual)
- `exec_error`:执行抛异常(附 traceback)

特殊:若工具签名首参为 `g`,自动注入 `g=globals()`(支持元编程类工具)。

### 10.3 自纠错 chat 循环

```python
while finish_reason == "tool_calls":
    并行 gather 所有 tool_calls
    if any(_tool_status=="error") and retry < max_tool_retries:
        追加一条 user 消息: "【系统通知】你之前调用的工具执行失败...请分析错误原因并修正后重新调用"
        retry += 1
    继续下一轮
```

### 10.4 batch 并发

`tool_calling/batch.py`:`asyncio.Semaphore(max_concurrent=20)` 限并发,通过 `(idx, result)` 元组排序保证返回顺序与输入一致。

## 十一、本地评测器(`local/`)

`LocalEvaluator(problem_path, user_code_path, llm_config, on_event=None)` 是**云端流程的本机镜像**:

- `_load_user_files`:读 `solution.py` + `requirements.txt`(默认 `openai\n`),支持文件或目录
- `_validate_config`:检查 `solve_attr_name`/`adapter_preset`/`sandbox/user_adapter.py` 存在
- `_setup_sigint_guard`:第二次 Ctrl+C 时强制退出,阻止清理被打断
- `evaluate()`:复用 `_SandboxPairSession`、`PairSessionDeps`、`run_pair_protocol_router`,只替换:
  - env builder:本地填 LLM key 而非 gateway token
  - problem loader:动态 import `<problem>/config.py`,取 `*Config` 类
  - artifact builder:扫描 `<problem>/sandbox/` 直接构造 bytes 字典
  - 注入 3 个 helper 文件:`_agent_wrapper.py`、`_user_bridge.py`、`eval_runtime/problem_adapter.py`
- `evaluate_stream()`:背景线程驱动,主线程消费 `Queue[EvalEvent]`
- `EvalEventType`:`CASE_START / CASE_END / OBSERVATION / ACTION / JUDGE_LOG / USER_LOG / PROGRESS / ERROR`,每种自带 `__str__` 格式化
- `TerminalVisualizer`:`grouped` 或 `interleaved` 模式,ANSI 彩色 + 字数截断 + 线程锁,适合 CLI 演示

通过条件 `min_passed_cases`(若 PhaseConfig 设置)或"全部 case 通过"判定整体 pass。

## 十二、整体设计亮点

1. **双沙箱隔离**:judge 与 user 物理分离,judge 永远拿不到用户私有数据,用户也看不到题目内部 API
2. **协议归一**:无论 single/multi-agent,统一走 `MessageType` 状态机,扩展只需注册新 type
3. **错误兜底完整**:`_coerce_action_message` + `is_likely_mle_exit` + `attach_case_history` + `parse_case_result` 强制把所有异常路径都归一为合法 case_result
4. **Protocol vs ABC 双契约**:解决 sandbox 内动态 import 模块路径不一致导致的 isinstance 失败
5. **Gateway 限流可叠加**:`aggregate_limit × num_cases`,既能精准控总量又方便单 case 写代码
6. **本地零差异复现**:`LocalEvaluator` 与云上共享 90% 代码路径,几乎不存在"本地能跑云上炸"的环境差
